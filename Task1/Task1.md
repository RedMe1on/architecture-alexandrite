
# Анализ архитектуры системы "Александрит"

## Существующие и потенциальные проблемные места:

1. **Перегрузка системы заказов**:
   - **Проблема:** Внедрение API для обработки заказов от других продавцов привело к значительному увеличению нагрузки, в том числе на систему обработки заказов.
   - **Причина:** Увеличение числа заказов при сохранении текущих процессов обработки привело к их затруднению. Основной нагрузкой является **интернет-магазин** и **Shop API**, которые не справляются с увеличением объёма информации и запросов.

2. **Проблемы с обработкой и статусами заказов**:
   - **Проблема:** Клиенты жалуются на просроченные заказы. В частности, **операторы** не могут своевременно получить актуальную информацию о заказах.
   - **Причина:** Логика отображения заказов на первой странице MES не оптимизирована для большого объёма данных. Это затрудняет оперативную работу операторов, что влияет на своевременность выполнения заказов.

3. **Неоптимизированная работа с очередями сообщений**:
   - **Проблема:** Постоянные жалобы от **API users** на невыдачу заказов.
   - **Причина:** Очереди сообщений **RabbitMQ** могут быть перегружены большим количеством входящих запросов, что препятствует своевременному передаче информации в систему.

4. **Низкая производительность фронт‑энд компонентов**:
   - **Проблема:** **CRM** и **Internet Shop** не справляются с высоким числом клиентов, что ведет к задержкам при взаимодействии с пользователями.
   - **Причина:** Низкая производительность сервисов, которые взаимодействуют с пользователями на веб‑страницах, включая **Vue** и **SpringBoot**, не оптимизированы для роста нагрузки.

5. **Отсутствие масштабируемости на уровне инфраструктуры**:
   - **Проблема:** Масштабируемость системы не учтена на уровне базы данных и хранения файлов.
   - **Причина:** **Shop DB** и **MES DB** не могут эффективно обрабатывать возросшее количество данных, что приводит к задержкам и сбоям.

## Инициативы для устранения проблем:

1. **Оптимизация базы данных**:
   - Масштабировать **Shop DB** и **MES DB**, улучшив их производительность с помощью репликации и шардирования. Это позволит системе эффективно работать с растущими объёмами данных.

2. **Оптимизация очередей сообщений**:
   - Внедрить стратегии для балансировки нагрузки в **RabbitMQ**, например, использование **расширенных очередей**, распределённых по топикам, для обработки разных типов сообщений с заказами.

3. **Повышение производительности фронтенда и бэкенда**:
   - Улучшить компоненты **CRM** и **Internet Shop**, внедрив технологии кэширования для снижения нагрузки на серверы. Рассмотреть использование **Content Delivery Network (CDN)** для ускорения работы с мультимедийным контентом.

4. **Оптимизация работы с заказами в MES**:
   - Переписать логику отображения заказов в **MES**, внедрив **кэширование данных** и улучшив фильтрацию и пагинацию на первой странице, чтобы ускорить загрузку.

5. **Модернизация API и интеграции с внешними сервисами**:
   - Улучшить **API** для работы с внешними пользователями, улучшить обработку ошибок и увеличить устойчивость системы при большом количестве запросов от сторонних продавцов.

6. **Внедрение системы мониторинга и логирования**:
   - Реализовать продвинутое **мониторинг приложений** для отслеживания производительности и раннего выявления проблем с заказами, очередями сообщений и загрузкой сервисов.

## Целевая архитектура через полгода:

Через полгода архитектура должна быть **масштабируемой и устойчивой** к росту нагрузки. Ожидается внедрение:

- **Гибкого масштабирования базы данных** с использованием **шардирования** и **репликации**.
- **Оптимизированной очереди сообщений** с балансировкой нагрузки.
- **Высокопроизводительных фронтенд и бэкенд компонентов** с использованием **кэширования** и **CDN**.
- **Автоматической балансировки нагрузки** на уровне API и интеграции с внешними сервисами.

Эта архитектура обеспечит стабильную работу системы в условиях растущих объёмов заказов и улучшит пользовательский опыт.
