## Мотивация

Для компании "Александрит" важно внедрить систему мониторинга по нескольким причинам:

1. **Раннее обнаружение проблем**: В условиях увеличивающейся нагрузки важно отслеживать каждый компонент системы, чтобы выявить возможные узкие места (например, базы данных или очереди сообщений) и оперативно реагировать на сбои, что минимизирует время простоя.

2. **Улучшение качества обслуживания клиентов**: Мониторинг поможет выявить задержки или сбои в процессах обработки заказов, что даст возможность сократить количество жалоб от клиентов и повысить удовлетворенность.

3. **Повышение эффективности работы сотрудников**: С помощью мониторинга можно отслеживать производительность операторов и других пользователей системы, что позволит улучшить внутренние процессы и повысить эффективность.

4. **Подготовка к росту**: Система мониторинга позволит подготовиться к масштабированию и эффективному управлению растущими объемами данных и запросов. Это обеспечит компанию стабильностью при расширении бизнеса.

5. **База для дальнейшей оптимизации**: Мониторинг станет основой для всех дальнейших улучшений в системе, таких как оптимизация работы сервисов, добавление кэширования и повышение отказоустойчивости. Без мониторинга невозможно точно понять, какие компоненты требуют улучшений.

6. **Понимание поведения внешних пользователей API**: Важно не только мониторить внутренние компоненты, но и отслеживать поведение внешних пользователей через API. Это даст возможность выявить пиковые нагрузки, проблемы с интеграциями и потенциальные угрозы, такие как перегрузки или атаки на сервисы.

## Выбор подхода к мониторингу

Для эффективного мониторинга системы компании "Александрит" я рекомендую использовать **подход "Четырех золотых сигналов"** для критичных сервисов, таких как базы данных, очереди сообщений, а также **подход RED** для микросервисов (например, API, CRM, интернет-магазин и MES).

### 1. Четыре золотых сигнала:
Этот подход хорошо подходит для **критичных сервисов** и **инфраструктуры**, таких как:
- **Shop DB**
- **MES DB**
- **Messages Queue**
- **3D files storage**

Подход **"Четырех золотых сигналов"** включает:
- **Latency (Задержка):** Время отклика базы данных или очереди сообщений.
- **Traffic (Трафик):** Количество запросов или сообщений, переданных через сервис.
- **Errors (Ошибки):** Частота ошибок или сбоев.
- **Saturation (Насыщенность):** Использование ресурсов, таких как CPU, память, хранилище и т.д.

### 2. Подход RED:
Этот подход подходит для **микросервисов** и компонентов, таких как:
- **Internet Shop**
- **Shop API**
- **CRM**
- **MES API**

Подход **RED** включает:
- **Rate (Частота запросов):** Количество входящих запросов в микросервис.
- **Errors (Ошибки):** Частота ошибок, возникающих в микросервисах.
- **Duration (Продолжительность):** Время выполнения запросов.

Этот подход позволит оптимизировать взаимодействие между сервисами и вовремя реагировать на перегрузки или сбои.

### 3. Комбинированный подход:
Использование обоих подходов для разных частей системы позволит создать **гибкую и детализированную систему мониторинга**, которая будет отслеживать как инфраструктуру, так и пользовательские сервисы. Это обеспечит быструю диагностику проблем на разных уровнях и поможет компании эффективно масштабироваться.

## План действий

## 1. **Создание инстанса базы данных для временных рядов**

### Задача:
Создать новый инстанс **Prometheus** для хранения метрик мониторинга. Эта база будет использоваться для хранения данных о производительности, трафике и ошибках, таких как задержки, количество запросов, загрузка CPU и памяти и т.д.
  
### Где метрики внедряются:
- Все метрики, связанные с нагрузкой и производительностью сервисов, будут собираться и храниться в выбранной базе данных.
  
---

## 2. **Внедрение системы мониторинга с использованием выбранных метрик**

### Задача:
Настроить сбор и агрегацию метрик для ключевых компонентов системы, таких как **RabbitMQ**, **Shop API**, **MES API**, **CRM API**, и базы данных.

### Шаги:
- **RabbitMQ**:
  - Внедрить мониторинг метрик, таких как:
    - **Количество сообщений в Dead-letter exchange**.
    - **Количество сообщений в полете (in-flight)**.
    - **Задержки при обработке сообщений**.
    - **Ошибки при обработке сообщений**.
  - Использовать **Prometheus** с экспортерами для **RabbitMQ**.
  
- **Shop API, CRM API, MES API**:
  - Внедрить мониторинг метрик API:
    - **Частота запросов (RPS)** для каждого сервиса.
    - **Ошибки**: количество ошибок 500 и другие статусы.
    - **Время отклика (latency)** для каждого сервиса.
    - **Использование CPU и памяти** для каждого сервиса.
  - Использовать **Prometheus** для сбора метрик и **Grafana** для визуализации.

- **Shop DB и MES DB**:
  - Внедрить мониторинг метрик базы данных:
    - **Количество соединений** с БД.
    - **Размер БД**.
    - **Использование памяти**.
    - **Задержка при запросах**.
  - Использовать **Prometheus** с экспортерами для **MySQL/PostgreSQL** или **Telegraf** для мониторинга БД.

### Где метрики внедряются:
- **RabbitMQ**: Внедряются метрики для отслеживания обработки сообщений (Dead-letter, in-flight, ошибки и задержки).
- **API сервисы**: Внедряются метрики для отслеживания трафика, ошибок, задержек и использования ресурсов для **Shop API**, **CRM API**, и **MES API**.
- **Базы данных**: Внедряются метрики для отслеживания использования ресурсов, производительности и задержек для **Shop DB** и **MES DB**.

## 3. **Настройка алертов и уведомлений**

### Задача:
Настроить систему алертов для критичных метрик, чтобы оперативно реагировать на перегрузки и сбои в системе.

### Шаги:
- **RabbitMQ**:
  - Настроить алерты на **перегрузку очередей**: если количество сообщений в очереди превышает пороговое значение (например, 80% от максимальной емкости), настроить уведомление.
  - Настроить алерты на **задержки при обработке сообщений**.
  
- **Shop API, CRM API, MES API**:
  - Настроить алерты на **высокую загрузку CPU** (если использование CPU превышает 85%).
  - Настроить алерты на **высокую нагрузку на память** (если использование памяти превышает 90%).
  - Настроить алерты на **ошибки 500** (если доля ошибок превышает 5% от общего числа запросов за 10 минут).

- **Shop DB и MES DB**:
  - Настроить алерты на **перегрузку соединений** (если количество соединений превышает порог).
  - Настроить алерты на **высокое использование памяти** или **большой размер базы данных**.

- **Интеграция с Alertmanager**:
  - Настроить интеграцию с **Alertmanager** для отправки уведомлений через Slack, email или другие каналы.
  
### Где метрики внедряются:
- **RabbitMQ**: Алерты на перегрузку очередей и задержки.
- **API сервисы**: Алерты на использование ресурсов (CPU, память) и ошибки.
- **Базы данных**: Алерты на нагрузку и использование памяти.

# Дополнительное задание: Показатели насыщенности

### Показатели насыщенности, которые должны отслеживаться:

1. **CPU и Memory Utilisation** для API и БД:
   - **Пороговое значение**: 85% для CPU и 90% для памяти.
     - **Почему эти значения**: Если нагрузка на CPU или память превышает 85-90%, это может вызвать деградацию производительности системы, замедление работы и возможно её сбой. Эти пороги помогут нам выявить проблемы до того, как они начнут сильно влиять на пользователей.

   - **Что происходит при превышении порога**:
     - **Автоматическое масштабирование**: Если значения превышают порог, система должна автоматически создать новые инстансы для сервисов, чтобы справиться с нагрузкой.
     - **Уведомление**: Если система не может масштабироваться (например, из-за нехватки ресурсов), должно быть отправлено уведомление ответственным сотрудникам через Slack или email.
     - **Создание тикета в системе**: Если нагрузка продолжает оставаться высокой в течение 10 минут, автоматически создается тикет в системе поддержки для дальнейшего анализа.

2. **Количество сообщений в очереди RabbitMQ**:
   - **Пороговое значение**: Если в очереди более 80% сообщений по сравнению с её максимальной емкостью.
     - **Почему эти значения**: Очереди RabbitMQ могут стать узким местом, если слишком много сообщений ждут обработки. Это увеличивает задержки и снижает производительность.

   - **Что происходит при превышении порога**:
     - **Добавление новых потребителей**: Если очередь переполняется, система автоматически добавляет дополнительные потребители RabbitMQ для более быстрой обработки сообщений.
     - **Оповещение о проблемах с очередью**: Уведомление отправляется в случае, если количество сообщений превышает пороговое значение в течение 5 минут.

3. **Количество ошибок HTTP 500 для API**:
   - **Пороговое значение**: Более 5% от общего числа запросов в течение 10 минут.
     - **Почему эти значения**: Частые ошибки 500 указывают на проблемы на сервере, которые могут повлиять на работоспособность всего приложения.

   - **Что происходит при превышении порога**:
     - **Автоматическое увеличение ресурсов**: Если количество ошибок превышает порог, система должна автоматически масштабировать количество серверов, обрабатывающих API.
     - **Сообщение в систему мониторинга**: Если ошибка не устраняется в течение 10 минут, нужно создать тикет для команды поддержки.
