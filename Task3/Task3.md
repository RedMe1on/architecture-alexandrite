## Мотивация

Внедрение системы трейсинга в систему компании "Александрит" важно по следующим причинам:

1. **Обнаружение узких мест и зависших заказов**:
   - Система трейсинга позволяет детально отслеживать путь каждого запроса в системе, что помогает найти места, где заказы могут «сломаться» или зависнуть. Это позволяет компании быстрее реагировать на сбои и устранять их.

2. **Улучшение производительности и стабильности системы**:
   - С помощью трейсинга можно определить, где происходят задержки в обработке заказов и в системе в целом, что позволит оптимизировать процессы и повысить стабильность.

3. **Повышение качества обслуживания клиентов**:
   - Благодаря трейсингу, можно быстро выявить, на каком этапе заказа возникла проблема, и оперативно уведомить клиента о статусе его заказа. Это улучшит репутацию компании.

4. **Предотвращение отказов и задержек**:
   - Возможность анализировать каждый этап выполнения заказа позволяет вовремя устранять проблемы до того, как они приведут к полному сбою.

## Предлагаемое решение

Для внедрения трейсинга, предлагаю использовать **OpenTelemetry**, который является универсальным инструментом для сбора данных о трассировке, метках и логах. Этот инструмент позволяет интегрировать трейсинг в различные компоненты системы без значительных изменений в коде.

### Компоненты, которые нужно покрыть трейсингом:

1. **RabbitMQ**: Трейсинг должен отслеживать сообщение от создания до обработки, чтобы можно было видеть, где оно "зависает".
2. **Shop API, CRM API, MES API**: Для каждого API запроса нужно отслеживать путь запроса, время отклика и ошибки.
3. **Shop DB и MES DB**: Необходимо отслеживать время обработки запросов к базам данных, чтобы понять, где возникают задержки.
4. **Интернет-магазин**: Трейсинг для клиентских операций, таких как добавление товара в корзину или оформление заказа.
5. **CRM**: Для анализа работы с клиентами и отслеживания всех этапов обработки заказов.


### Архитектура решения

Будут добавлены следующие компоненты:

1. **OpenTelemetry SDK (в каждом сервисе)**  
   - Встраивается в:
     - Shop API  
     - CRM API  
     - MES API  
     - обработчики RabbitMQ  
   - Отвечает за генерацию trace/span

2. **OpenTelemetry Collector**  
   - Централизованный сбор трассировок  
   - Агрегация и маршрутизация данных  
   - Снижение нагрузки на сервисы

3. **Система хранения и визуализации трассировок**  
   - Jaeger
   - Позволяет анализировать цепочки запросов

## Компромиссы

Трейсинг не везде даст одинаковую пользу, и его внедрение имеет ограничения.

### Где трейсинг может не дать заметного эффекта
1. **Статические или почти неизменяемые компоненты**  
   Для простых компонентов без сложной логики и без критичных цепочек вызовов ценность трейсинга ниже, чем у обычных метрик и логов.

2. **Очень короткие локальные операции**  
   Если операция выполняется внутри одного процесса и не вызывает внешние зависимости, трейсинг может дать мало дополнительной информации по сравнению с профилированием и application-логами.

3. **Редкие фоновые задачи без жёсткого SLA**  
   Для неважных служебных джобов достаточно базового логирования и метрик успешности выполнения.

### Где внедрение затруднено или слишком дорого
1. **Проприетарные или плохо расширяемые системы**  
   Если часть MES/CRM или интеграционного ПО не поддерживает OpenTelemetry или не даёт внедрить middleware/agent, потребуется дорогая доработка или адаптер.

2. **Старые асинхронные интеграции**  
   Если сообщения в очередях не содержат correlation-id и trace context, потребуется изменить формат событий и доработать всех потребителей.

3. **Большой объём трассировок**  
   При полном трейсинге всех заказов и всех фоновых операций резко вырастут объёмы хранения и стоимость эксплуатации. Поэтому потребуется sampling и отдельные правила для критичных заказов.

4. **Высоконагруженные участки с малой ценностью отдельных span**  
   На очень «шумных» участках избыточная детализация может повысить накладные расходы на CPU, сеть и storage без сопоставимой пользы.

### Принятые компромиссы
- Для B2C и B2B заказов используется **приоритетный трейсинг критичного пути заказа**.
- Для высокочастотных операций включается **sampling**, но для ошибок и долгих запросов трассировки сохраняются всегда.
- Для неподконтрольных или закрытых компонентов допускается **частичный трейсинг** через границы API, логи и correlation-id.

## Безопасность

Система трейсинга должна быть защищена как внутри компании, так и на внешних границах.

### Меры защиты внутри компании
1. **Аутентификация через корпоративный SSO**  
   Доступ к UI Jaeger получают только сотрудники компании с актуальной учётной записью.

2. **RBAC / ролевой доступ**  
   - Роль **Поддержка**: просмотр трассировок и дешбордов  
   - Роль **Разработка**: просмотр детальных span и технических атрибутов  
   - Роль **Администратор платформы**: настройка collector, retention, sampling

3. **Сегментация сети**  
   OpenTelemetry Collector, storage и UI размещаются во внутреннем сегменте сети. Прямой доступ извне отсутствует.

4. **Шифрование трафика**  
   Передача trace-данных между сервисами, collector и backend выполняется по TLS/mTLS.

5. **Аудит доступа**  
   Все входы в UI и административные действия логируются.

### Меры защиты данных
1. **Запрет на отправку чувствительных данных в span**  
   В trace не должны попадать:
   - пароли
   - токены
   - полные персональные данные клиентов
   - номера платёжных инструментов
   - содержимое файлов

2. **Маскирование и редактирование атрибутов**  
   Допускается хранить только:
   - `order_id`
   - `partner_id`
   - внутренний идентификатор клиента
   - технические коды ошибок
   - статус заказа

3. **Retention policy**  
   Хранить трассировки ограниченное время, например 7–30 дней в зависимости от категории данных.

4. **Ограничение внешнего доступа**  
   Если доступ извне всё же нужен для подрядчиков или внешней поддержки, он должен идти только через VPN, корпоративный IdP и отдельные роли с временными правами.
